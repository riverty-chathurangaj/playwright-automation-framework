SECTION 1

Executive Summary

This document presents the architecture for Testonaut GL — an intelligent, AI-augmented test automation framework designed to ensure the quality, reliability, and contract compliance of the GL (General Ledger) Service API.

The GL Service API is a financial-domain .NET application that requires rigorous testing across functional, schema, contract, and security dimensions. The test framework is intentionally decoupled from the development project, operating as an independent black-box test suite that communicates exclusively through the API's HTTP interface.

Key Decision

The framework is built in TypeScript rather than .NET. This decoupling enables unrestricted negative testing (sending malformed payloads that .NET's type system would reject at compile time), provides access to the richest AI/ML ecosystem, and ensures the test suite remains an independent quality gate.

Framework Objectives

🧪

Comprehensive API Testing

Full coverage of GL endpoints including positive flows, negative scenarios, boundary conditions, and security edge cases — all in Gherkin syntax.

📐

Schema & Contract Validation

Automated JSON Schema validation and consumer-driven contract testing with drift detection to catch breaking changes before deployment.

📨

RabbitMQ Message Testing

Full message lifecycle validation — publish, consume, assert on message content, headers, routing keys. Dead-letter queue verification, message ordering, and event-driven flow testing.

🗄️

Database Validation Testing

Direct database assertions to verify data persistence, referential integrity, transaction correctness, audit trails, and GL-specific balancing rules at the data layer.

🤖

AI-Powered Intelligence

AI-driven scenario generation from Jira tickets, intelligent test data synthesis, automated failure analysis, and self-healing test maintenance.

📊

Xray & Jira Integration

Seamless bi-directional integration with Xray Test Management — automated result publishing, traceability, and AI-enriched execution reports.

SECTION 2

Design Principles & Constraints

2.1 Why Not .NET

The decision to avoid .NET for the test framework is driven by multiple factors: a fundamental testing requirement for unrestricted payload manipulation, the fragile state of .NET BDD tooling, and the significantly larger testing ecosystem in TypeScript/JavaScript.

BDD Tooling Risk in .NET

.NET has only one viable BDD library: Reqnroll. Reqnroll is a fork of SpecFlow, which was a corporate-backed library maintained by Tricentis. SpecFlow was abandoned in December 2024, with its last release dating back to 2022 — a two-year gap without updates. Reqnroll is now maintained by a small group of open-source developers, carrying inherent sustainability risk.

.NET BDD Ecosystem Risk

SpecFlow: Corporate-backed by Tricentis → abandoned Dec 2024 (last release: 2022).
Reqnroll: Community fork of SpecFlow → maintained by a small open-source team with no corporate backing.
Update cadence: Both SpecFlow and Reqnroll have historically slow release cycles, causing compatibility issues with newer .NET versions and blocking framework upgrades.
Consequence: Dependence on Reqnroll creates a single point of failure for the entire BDD strategy.

Community & Ecosystem Gap

The testing community around .NET is significantly smaller than TypeScript/JavaScript. The BDD-specific .NET community is even smaller. In contrast, the TS/JS testing ecosystem (Playwright, Cypress, Jest, Vitest, playwright-bdd) evolves rapidly with frequent releases, large contributor bases, and broad plugin ecosystems. This translates directly into faster issue resolution, better documentation, and more integration options.

Technical Advantages

Constraint .NET Limitation TypeScript Advantage
Type Safety Override C# will not compile if you assign a string to a decimal field. Requires dynamic types or raw JSON everywhere. TypeScript's structural typing and any type allow sending "abc" where a number is expected — essential for negative testing.
Model Independence Without referencing the dev project's models, you'd rebuild them — creating coupling risk. Request/response shapes are defined loosely. No compile-time coupling to the SUT.
Payload Mutation Corrupting payloads requires awkward workarounds with JObject or raw strings. Native object spread, optional chaining, and dynamic properties make mutation trivial.
AI Ecosystem Limited LLM SDK support. Semantic Kernel exists but smaller ecosystem. LangChain.js, Anthropic SDK, OpenAI SDK, HuggingFace — all TypeScript-first.
BDD Tooling Only Reqnroll (SpecFlow fork) — slow updates, .NET version compatibility lag, small community. playwright-bdd: active development, Playwright-native fixtures, parallel execution, rich ecosystem.
Community Velocity Small .NET testing community. BDD .NET community is minimal. Slow tooling evolution. Largest testing ecosystem in any language. Rapid tool evolution. Broad community support.

2.2 Black-Box Testing Philosophy

The framework treats the GL Service API as an opaque system. All interactions occur through the public HTTP interface, exactly as a real consumer would use it.

  • No internal dependency leakage — Tests remain valid even if internal implementation changes (ORM, database, domain models).
  • True consumer perspective — Tests validate what the API actually does, not what the code intends.
  • Unrestricted negative testing — Any payload shape can be sent without compile-time restrictions.
  • Independent deployment — The test suite can be versioned and executed independently of the application release cycle.

3.2 Technology Justification

Why Playwright over Axios/Supertest

Playwright's APIRequestContext provides built-in request/response tracing that captures every HTTP interaction with full headers and bodies — invaluable for debugging financial API issues. It handles cookie and authentication state management automatically, supports parallel execution with isolated contexts, and its retry mechanism is production-grade. If the framework ever needs UI testing (e.g., GL dashboard), Playwright is already in place.

Why Custom Contract Engine over Pact

While Pact is the industry standard for consumer-driven contracts, the GL domain has specific requirements benefiting from a tailored solution. Financial APIs need precision validation (e.g., ensuring balance fields have exactly 2 decimal places, not just "is a number"). The custom Ajv-based engine allows domain-specific validators: debit-credit balance checks, currency code validation (ISO 4217), and fiscal period boundary validation that Pact's generic matchers cannot express natively.

4.3 Core Components

API Client (api-client.ts)

The centralized HTTP client wraps Playwright's APIRequestContext providing a unified interface for all API interactions. It handles authentication header injection, request/response logging, response time measurement, and automatic retry for transient failures.

TypeScript
export class ApiClient {
  private context: APIRequestContext;
  private authManager: AuthManager;
  private logger: Logger;

  async request(method: string, endpoint: string, options?: RequestOptions) {
    const startTime = Date.now();
    const headers = await this.authManager.getAuthHeaders();

    const response = await this.context[method](endpoint, {
      headers: { ...headers, ...options?.headers },
      data: options?.body,
      params: options?.queryParams,
    });

    const duration = Date.now() - startTime;
    this.logger.info('API Request', {
      method, endpoint, status: response.status(), duration,
      requestBody: options?.body,
      responseBody: await response.json().catch(() => null)
    });

    return {
      status: response.status(),
      body: await response.json(),
      headers: response.headers(),
      duration
    };
  }
}

Payload Mutator (payload-mutator.ts)

The most critical component for negative testing. Takes a valid payload and applies configurable corruptions to specific fields, enabling comprehensive validation testing without .NET model constraints.

TypeScript
export type CorruptionType =
  | 'string-in-numeric' | 'overflow' | 'negative-amount'
  | 'null' | 'missing' | 'empty-string' | 'exceed-max-length'
  | 'sql-injection' | 'special-chars' | 'wrong-type-array'
  | 'wrong-type-object' | 'unicode';

export class PayloadMutator {
  static corruptField(
    payload: Record<string, any>,
    field: string,
    corruption: CorruptionType
  ): Record<string, any> {
    const mutated = structuredClone(payload);
    switch (corruption) {
      case 'string-in-numeric':  mutated[field] = 'not-a-number'; break;
      case 'overflow':           mutated[field] = Number.MAX_SAFE_INTEGER + 1; break;
      case 'negative-amount':    mutated[field] = -99999.99; break;
      case 'null':               mutated[field] = null; break;
      case 'missing':            delete mutated[field]; break;
      case 'empty-string':       mutated[field] = ''; break;
      case 'exceed-max-length':  mutated[field] = 'x'.repeat(10000); break;
      case 'sql-injection':      mutated[field] = "'; DROP TABLE--"; break;
      case 'special-chars':      mutated[field] = '<>&"\'\\'; break;
      case 'wrong-type-array':   mutated[field] = [1, 2, 3]; break;
      case 'wrong-type-object':  mutated[field] = { invalid: true }; break;
      case 'unicode':            mutated[field] = '日本語テスト'; break;
    }
    return mutated;
  }

6.1 JSON Schema Validation

Every API endpoint has a corresponding JSON Schema definition. The SchemaValidator uses Ajv to validate actual API responses against these schemas, catching structural regressions immediately.

JSON SCHEMA — gl-account.schema.json (actual)
{
  "$schema": "http://json-schema.org/draft-07/schema#",
  "$id": "gl-account",
  "title": "GLAccount",
  "type": "object",
  "properties": {
    "account":     { "type": ["string", "null"] },
    "description": { "type": ["string", "null"] }
  },
  "additionalProperties": true
}
Schema Design Note

Schemas use JSON Schema Draft-07 syntax. Nullable fields use ["type", "null"] — not OpenAPI's nullable: true. The $id field must match the name used in schemaValidator.validate('gl-account', item). Root type is always "object"; array responses use the common step each item in the response array should match schema.

The framework defines JSON schemas for each API response type. Schemas are auto-loaded by SchemaValidator from src/schemas/json-schemas/.

TypeScript — Schema Validator (schema-validator.ts)
import Ajv from 'ajv';
import addFormats from 'ajv-formats';
import { glob } from 'glob';
import { readFileSync } from 'fs';
import { join, basename } from 'path';

export class SchemaValidator {
  private ajv: Ajv;

  constructor() {
    this.ajv = new Ajv({ allErrors: true, strict: false });
    addFormats(this.ajv);
    this.loadSchemas(); // Auto-loads all *.schema.json files
  }

  private loadSchemas(): void {
    const schemaDir = join(__dirname, 'json-schemas');
    const files = glob.sync(join(schemaDir, '*.schema.json'));
    files.forEach(file => {
      const schema = JSON.parse(readFileSync(file, 'utf-8'));
      this.ajv.addSchema(schema, schema.$id);
    });
  }

  validate(schemaName: string, data: unknown): ValidationResult {
    const validate = this.ajv.getSchema(schemaName);
    if (!validate) throw new Error(`Schema "${schemaName}" not found`);
    const valid = validate(data) as boolean;
    return {
      valid,
      errors: validate.errors?.map(e => ({
        path: e.instancePath,
        message: e.message ?? '',
        keyword: e.keyword,
        params: e.params
      }))
    };
  }
}

SECTION 8

Database Validation Testing

While API response testing validates what the system returns, database testing validates what the system persists. For a financial GL service, data integrity at the storage layer is paramount — incorrect database state can propagate errors across the entire accounting system.

8.1 Database Testing Philosophy

Critical Rule

The test framework connects to the database as a read-only validator in most scenarios. Tests assert on data state but do not mutate it directly. Data mutations happen through the API under test. The exception is test data setup/teardown, which uses dedicated service accounts with scoped permissions.

ApproachWhen UsedExample
Post-API Assertion Verify API calls persisted data correctly After POST /journal-entries, assert the row exists with correct values
Snapshot Comparison Verify state transitions Capture account balance before and after posting; compare delta
Referential Integrity Verify foreign key relationships are maintained Journal entry references a valid account in the accounts table
Audit Trail Verification Verify compliance logging Every financial mutation has a corresponding audit_log entry with user, timestamp, and before/after values

Test Data Setup Pre-populate prerequisite data Insert reference accounts before testing journal entry posting

8.2 Database Client & Azure AD Auth

The framework connects to Azure SQL Database via Microsoft Entra ID passwordless authentication (@azure/identity → DefaultAzureCredential). No username/password required — the developer must be logged in via az login.

TypeScript — db-client.ts (actual implementation)
import knex, { Knex } from 'knex';
import { config } from '../core/config';

export class DatabaseClient {
  private db!: Knex;
  private connected: boolean = false;

  async connect(): Promise<void> {
    const dbConfig = config.database;
    this.db = knex({
      client: dbConfig.client,
      connection: this.buildConnectionConfig(dbConfig),
      pool: { min: 0, max: 5 },
      acquireConnectionTimeout: 120_000,
    });
    await this.db.raw('SELECT 1');
    this.connected = true;
  }

  private buildConnectionConfig(cfg: typeof config.database) {
    // Azure AD passwordless — knex maps `type` into tedious config
    if (cfg.client === 'mssql'
        && cfg.authType === 'azure-active-directory-default') {
      return {
        server: cfg.host,
        database: cfg.name,
        type: 'azure-active-directory-default',  // ← at root, NOT nested
        options: {
          encrypt: true,
          trustServerCertificate: false,
          requestTimeout: cfg.queryTimeout,
        },
      } as unknown as Knex.MsSqlConnectionConfig;
    }
    // Standard SQL auth fallback
    return {
      host: cfg.host, user: cfg.user, password: cfg.password,
      database: cfg.name, port: cfg.port,
    };
  }
}
Knex + Azure AD Gotcha

Set type: 'azure-active-directory-default' at the connection root level — NOT nested under authentication. Knex maps it internally via _generateConnection(). Nesting it under authentication causes knex to strip it.

TypeScript — Snapshot Manager
export class SnapshotManager {
  private snapshots: Map<string, any> = new Map();

  // Capture state before an operation
  async captureSnapshot(label: string, query: () => Promise<any>) {
    this.snapshots.set(label, await query());
  }

  // Compare with current state after an operation
  async compareSnapshot(label: string, query: () => Promise<any>) {
    const before = this.snapshots.get(label);
    const after = await query();
    return {
      before,
      after,
      changes: this.deepDiff(before, after),
      balanceDelta: after?.balance != null
        ? (after.balance - before.balance) : undefined
    };
  }
}

8.4 Test Data Lifecycle Management

Database test data must be carefully managed to ensure test isolation without corrupting shared environments. The CleanupManager tracks all data created during tests and removes it in reverse order after each scenario.

TypeScript — cleanup-manager.ts
export class CleanupManager {
  private createdRecords: { table: string; id: string; }[] = [];

  // Track records created during test setup
  trackRecord(table: string, id: string): void {
    this.createdRecords.push({ table, id });
  }

  // Teardown: remove all test data in reverse order (respects FK constraints)
  async cleanup(db: DatabaseClient): Promise<void> {
    for (const record of this.createdRecords.reverse()) {
      try {
        await db.query(`DELETE FROM ${record.table} WHERE id = ?`, [record.id]);
      } catch (error) {
        // Log but don't fail — cleanup is best-effort
        logger.warn('Cleanup failed', { table: record.table, id: record.id, error });
      }
    }
    this.createdRecords = [];
  }
}

Suggested Fix: Ensure the API date parser enforces ISO 8601 (yyyy-MM-dd) and returns 400 BadRequest with a descriptive message for invalid formats rather than a 500.
Related Patterns: Similar to accounting-month filter failures seen in previous runs.

TypeScript — Xray Reporter
export class XrayReporter {
  async publishResults(cucumberReport: CucumberJsonReport): Promise<void> {
    const token = await this.authenticate();

    const xrayPayload = {
      testExecutionKey: process.env.XRAY_EXECUTION_KEY,
      info: {
        summary: `GL API Tests - ${env} - ${new Date().toISOString()}`,
        description: `Automated BDD execution | Tags: ${tags}`,
        startDate: this.executionStartTime,
        finishDate: new Date().toISOString(),
        testEnvironments: [process.env.TEST_ENV],
        revision: process.env.GIT_SHA
      },
      tests: cucumberReport.features.flatMap(f =>
        f.elements.map(scenario => ({
          testKey: this.extractXrayKey(scenario.tags),
          status: this.mapStatus(scenario),
          comment: scenario.aiAnalysis || '',
          evidences: this.buildEvidences(scenario),
          steps: scenario.steps.map(step => ({
            status: this.mapStepStatus(step),
            comment: step.result.error_message || '',
            actualResult: step.result.duration + 'ms'
          }))
        }))
      )
    };

    await this.xrayClient.importExecution(xrayPayload);
  }
}

on:
  pull_request:
    branches: [main, develop]
  schedule:
    - cron: '0 2 * * *'       # Nightly at 2 AM UTC
  workflow_dispatch:
    inputs:
      environment:
        type: choice
        options: [dev, staging, uat]
      tags:
        description: 'Playwright project (e.g. smoke, regression, clients)'

  api-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }

      - name: Install dependencies
        run: npm ci

      - name: Run API Tests
        run: |
          npm run bdd:gen
          npx playwright test \
            --project="${{ inputs.tags || 'smoke' }}"
        env:
          BASE_URL: ${{ secrets[format('API_URL_{0}', inputs.environment)] }}
          AUTH_CLIENT_ID: ${{ secrets.AUTH_CLIENT_ID }}
          AUTH_CLIENT_SECRET: ${{ secrets.AUTH_CLIENT_SECRET }}
          DB_AUTH_TYPE: azure-active-directory-default

      - name: AI Failure Analysis
        if: failure()
        run: echo "AI analysis runs inline via _afterTestHook fixture during test execution"
        env:
          AI_ENABLED: 'true'
          AI_PROVIDER: azure-openai
          OPENAI_API_KEY: ${{ secrets.OPENAI_API_KEY }}
          OPENAI_ENDPOINT: ${{ secrets.OPENAI_ENDPOINT }}
          AI_MODEL: o4-mini

      - name: Publish to Xray
        if: always()
        run: npx ts-node src/support/xray-reporter.ts publish
        env:
          XRAY_CLIENT_ID: ${{ secrets.XRAY_CLIENT_ID }}
          XRAY_CLIENT_SECRET: ${{ secrets.XRAY_CLIENT_SECRET }}

      - name: Upload Reports
        if: always()
        uses: actions/upload-artifact@v4

          name: test-reports

SECTION 14

Risk Assessment & Mitigations

RiskProbabilityImpactMitigation
API documentation gaps — swagger.json may reference wrong component schemas High High Always verify response shape against a live API run. Swagger is a starting point only.
Test environment instability — shared environments with unreliable data High Medium Test data setup/teardown in fixtures. Unique identifiers per run. Dedicated test instances.
Azure AD token issues — expired or misconfigured Entra ID credentials Medium High DefaultAzureCredential with fallback chain. Ensure az login is current. CI uses managed identity.
Message timing / flakiness — async messages arriving late causing false failures High Medium Configurable timeouts (30s default). Polling with backoff for DB verification. Amount-specific matching.
Large table query timeouts — Data.Transaction scans without date filter Medium High Always pass CreatedDate cutoff from message sentTime. Query timeout set to 120s.
Team TypeScript adoption — .NET-focused team learning new stack Medium Medium Gherkin is language-agnostic. Steps are simple HTTP calls. Framework guide (AGENTS.md) and CLAUDE.md for AI-assisted development.
RabbitMQ test environment access — restricted broker access or shared queues Medium High Use exclusive temporary queues. Connection auto-reconnect via amqp-connection-manager. Document required AMQP permissions.
Schema/Contract maintenance — keeping definitions in sync with actual API High Medium JSON Schema validation in CI catches regressions immediately. @schema tag runs on every PR.
Summary

Testonaut GL is a comprehensive BDD API test automation framework built on Playwright and playwright-bdd. It covers all GL API domains with JSON Schema validation, tests RabbitMQ message flows end-to-end (publish → consume → database verification), and connects to Azure SQL via Entra ID passwordless auth. AI-powered failure analysis is available as an opt-in enhancement. Xray Cloud integration provides complete traceability from requirements to test execution.
