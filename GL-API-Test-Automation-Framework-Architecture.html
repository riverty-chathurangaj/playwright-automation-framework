SECTION 1

Executive Summary

This document presents the architecture for Testonaut GL — an intelligent, AI-augmented test automation framework designed to ensure the quality, reliability, and contract compliance of the GL (General Ledger) Service API.

The GL Service API is a financial-domain .NET application that requires rigorous testing across functional, schema, contract, and security dimensions. The test framework is intentionally decoupled from the development project, operating as an independent black-box test suite that communicates exclusively through the API's HTTP interface.

Key Decision

The framework is built in TypeScript rather than .NET. This decoupling enables unrestricted negative testing (sending malformed payloads that .NET's type system would reject at compile time), provides access to the richest AI/ML ecosystem, and ensures the test suite remains an independent quality gate.

Framework Objectives

🧪

Comprehensive API Testing

Full coverage of GL endpoints including positive flows, negative scenarios, boundary conditions, and security edge cases — all in Gherkin syntax.

📐

Schema & Contract Validation

Automated JSON Schema validation and consumer-driven contract testing with drift detection to catch breaking changes before deployment.

📨

RabbitMQ Message Testing

Full message lifecycle validation — publish, consume, assert on message content, headers, routing keys. Dead-letter queue verification, message ordering, and event-driven flow testing.

🗄️

Database Validation Testing

Direct database assertions to verify data persistence, referential integrity, transaction correctness, audit trails, and GL-specific balancing rules at the data layer.

🤖

AI-Powered Intelligence

AI-driven scenario generation from Jira tickets, intelligent test data synthesis, automated failure analysis, and self-healing test maintenance.

📊

Xray & Jira Integration

Seamless bi-directional integration with Xray Test Management — automated result publishing, traceability, and AI-enriched execution reports.

SECTION 2

Design Principles & Constraints

2.1 Why Not .NET

The decision to avoid .NET for the test framework is driven by a fundamental testing requirement: the ability to send invalid, malformed, and type-violating payloads to the API. This is essential for financial software where input validation must be bulletproof.

Constraint .NET Limitation TypeScript Advantage
Type Safety Override C# will not compile if you assign a string to a decimal field. Requires dynamic types or raw JSON everywhere. TypeScript's structural typing and any type allow sending "abc" where a number is expected — essential for negative testing.
Model Independence Without referencing the dev project's models, you'd rebuild them — creating coupling risk. Request/response shapes are defined loosely. No compile-time coupling to the SUT.
Payload Mutation Corrupting payloads requires awkward workarounds with JObject or raw strings. Native object spread, optional chaining, and dynamic properties make mutation trivial.
AI Ecosystem Limited LLM SDK support. Semantic Kernel exists but smaller ecosystem. LangChain.js, Anthropic SDK, OpenAI SDK, HuggingFace — all TypeScript-first.
BDD Tooling SpecFlow is mature but has a smaller plugin ecosystem. Cucumber.js has the largest BDD ecosystem with rich formatters and integrations.

2.2 Black-Box Testing Philosophy

The framework treats the GL Service API as an opaque system. All interactions occur through the public HTTP interface, exactly as a real consumer would use it.

  • No internal dependency leakage — Tests remain valid even if internal implementation changes (ORM, database, domain models).
  • True consumer perspective — Tests validate what the API actually does, not what the code intends.
  • Unrestricted negative testing — Any payload shape can be sent without compile-time restrictions.
  • Independent deployment — The test suite can be versioned and executed independently of the application release cycle.

3.2 Technology Justification

Why Playwright over Axios/Supertest

Playwright's APIRequestContext provides built-in request/response tracing that captures every HTTP interaction with full headers and bodies — invaluable for debugging financial API issues. It handles cookie and authentication state management automatically, supports parallel execution with isolated contexts, and its retry mechanism is production-grade. If the framework ever needs UI testing (e.g., GL dashboard), Playwright is already in place.

Why Custom Contract Engine over Pact

While Pact is the industry standard for consumer-driven contracts, the GL domain has specific requirements benefiting from a tailored solution. Financial APIs need precision validation (e.g., ensuring balance fields have exactly 2 decimal places, not just "is a number"). The custom Ajv-based engine allows domain-specific validators: debit-credit balance checks, currency code validation (ISO 4217), and fiscal period boundary validation that Pact's generic matchers cannot express natively.

4.3 Core Components

API Client (api-client.ts)

The centralized HTTP client wraps Playwright's APIRequestContext providing a unified interface for all API interactions. It handles authentication header injection, request/response logging, response time measurement, and automatic retry for transient failures.

TypeScript
export class ApiClient {
  private context: APIRequestContext;
  private authManager: AuthManager;
  private logger: Logger;

  async request(method: string, endpoint: string, options?: RequestOptions) {
    const startTime = Date.now();
    const headers = await this.authManager.getAuthHeaders();

    const response = await this.context[method](endpoint, {
      headers: { ...headers, ...options?.headers },
      data: options?.body,
      params: options?.queryParams,
    });

    const duration = Date.now() - startTime;
    this.logger.info('API Request', {
      method, endpoint, status: response.status(), duration,
      requestBody: options?.body,
      responseBody: await response.json().catch(() => null)
    });

    return {
      status: response.status(),
      body: await response.json(),
      headers: response.headers(),
      duration
    };
  }
}

Payload Mutator (payload-mutator.ts)

The most critical component for negative testing. Takes a valid payload and applies configurable corruptions to specific fields, enabling comprehensive validation testing without .NET model constraints.

TypeScript
export type CorruptionType =
  | 'string-in-numeric' | 'overflow' | 'negative-amount'
  | 'null' | 'missing' | 'empty-string' | 'exceed-max-length'
  | 'sql-injection' | 'special-chars' | 'wrong-type-array'
  | 'wrong-type-object' | 'unicode';

export class PayloadMutator {
  static corruptField(
    payload: Record<string, any>,
    field: string,
    corruption: CorruptionType
  ): Record<string, any> {
    const mutated = structuredClone(payload);
    switch (corruption) {
      case 'string-in-numeric':  mutated[field] = 'not-a-number'; break;
      case 'overflow':           mutated[field] = Number.MAX_SAFE_INTEGER + 1; break;
      case 'negative-amount':    mutated[field] = -99999.99; break;
      case 'null':               mutated[field] = null; break;
      case 'missing':            delete mutated[field]; break;
      case 'empty-string':       mutated[field] = ''; break;
      case 'exceed-max-length':  mutated[field] = 'x'.repeat(10000); break;
      case 'sql-injection':      mutated[field] = "'; DROP TABLE--"; break;
      case 'special-chars':      mutated[field] = '<>&"\'\\'; break;
      case 'wrong-type-array':   mutated[field] = [1, 2, 3]; break;
      case 'wrong-type-object':  mutated[field] = { invalid: true }; break;
      case 'unicode':            mutated[field] = '日本語テスト'; break;
    }
    return mutated;
  }

5.2 Step Definition Strategy

Common, reusable steps handle generic HTTP operations and validation. Domain-specific steps handle GL business logic. This prevents duplication and ensures consistency.

import Ajv from 'ajv';
import addFormats from 'ajv-formats';

export class SchemaValidator {
  private ajv: Ajv;

  constructor() {
    this.ajv = new Ajv({ allErrors: true, strict: false });
    addFormats(this.ajv);
    this.loadSchemas(); // Auto-loads all .schema.json files
  }

  validate(schemaName: string, data: unknown): ValidationResult {
    const validate = this.ajv.getSchema(schemaName);
    const valid = validate(data);
    return {
      valid,
      errors: validate.errors?.map(e => ({
        path: e.instancePath,
        message: e.message,
        keyword: e.keyword,
        params: e.params
      }))
    };
  }

  // Contract validation: consumer expectations
  validateContract(contractName: string, response: any): ContractResult {
    const contract = this.loadContract(contractName);
    return {
      schemaValid: this.validate(contract.schema, response).valid,
      requiredFieldsPresent: this.checkRequiredFields(contract, response),
      fieldTypesMatch: this.checkFieldTypes(contract, response),
      enumValuesValid: this.checkEnumConstraints(contract, response),
      businessRulesValid: this.checkBusinessRules(contract, response)
    };
  }
}

7.2 Queue Client & Consumer Harness

The RabbitClient manages AMQP connections and channels, while the ConsumerHarness provides a high-level interface for subscribing to queues, collecting messages with configurable timeouts, and asserting on received messages.

TypeScript — rabbit-client.ts
import amqplib, { Connection, Channel, ConsumeMessage } from 'amqplib';

export class RabbitClient {
  private connection: Connection;
  private channel: Channel;

  async connect(): Promise<void> {
    this.connection = await amqplib.connect(config.rabbitmq.url);
    this.channel = await this.connection.createChannel();
  }

  async createTestQueue(exchange: string, routingKey: string): Promise<string> {
    const queueName = `test-${exchange}-${Date.now()}`;
    await this.channel.assertQueue(queueName, {
      exclusive: true,      // Auto-delete when test disconnects
      durable: false,
      arguments: { 'x-expires': 300000 } // 5-min TTL safety net
    });
    await this.channel.bindQueue(queueName, exchange, routingKey);
    return queueName;
  }

  async disconnect(): Promise<void> {
    await this.channel?.close();
    await this.connection?.close();
  }
}
TypeScript — consumer-harness.ts
export class ConsumerHarness {
  private collectedMessages: CollectedMessage[] = [];
  private rabbitClient: RabbitClient;

  async startListening(exchange: string, routingKey: string): Promise<void> {
    const queueName = await this.rabbitClient.createTestQueue(exchange, routingKey);
    await this.rabbitClient.channel.consume(queueName, (msg) => {
      if (msg) {
        this.collectedMessages.push({
          content: JSON.parse(msg.content.toString()),
          headers: msg.properties.headers,
          routingKey: msg.fields.routingKey,
          exchange: msg.fields.exchange,
          timestamp: new Date(),
          messageId: msg.properties.messageId,
          correlationId: msg.properties.correlationId,
          contentType: msg.properties.contentType
        });
        this.rabbitClient.channel.ack(msg);
      }
    });
  }

  // Wait for expected messages with timeout
  async waitForMessages(
    count: number,
    timeoutMs: number = 10000
  ): Promise<CollectedMessage[]> {
    const start = Date.now();
    while (this.collectedMessages.length < count) {
      if (Date.now() - start > timeoutMs) {
        throw new Error(
          `Timeout: expected ${count} messages, received ${this.collectedMessages.length}`
        );
      }
      await new Promise(r => setTimeout(r, 100));
    }
    return this.collectedMessages;
  }

  // Assert no unexpected messages arrived
  async assertNoMessages(waitMs: number = 3000): Promise<void> {
    await new Promise(r => setTimeout(r, waitMs));
    if (this.collectedMessages.length > 0) {
      throw new Error(
        `Expected no messages but received ${this.collectedMessages.length}`
      );
    }
  }

  reset(): void { this.collectedMessages = []; }
}

The framework uses Knex.js as the database client — a SQL query builder that supports SQL Server, PostgreSQL, MySQL, and SQLite. This gives flexibility regardless of which database the GL service uses.

import knex, { Knex } from 'knex';

export class DatabaseClient {
  private db: Knex;

  constructor() {
    this.db = knex({
      client: config.database.client,  // 'mssql', 'pg', 'mysql2'
      connection: {
        host: config.database.host,
        port: config.database.port,
        user: config.database.user,
        password: config.database.password,
        database: config.database.name,
      },
      pool: { min: 1, max: 5 }
    });
  }

  // Generic query execution
  async query<T>(sql: string, params?: any[]): Promise<T[]> {
    const result = await this.db.raw(sql, params);
    return result.rows || result; // Normalize across dialects
  }

  // Domain-specific helper: Get account by code
  async getAccountByCode(accountCode: string) {
    return this.db('gl_accounts').where({ account_code: accountCode }).first();
  }

  // Domain-specific helper: Get journal entries for account
  async getJournalEntriesForAccount(accountCode: string, limit = 50) {
    return this.db('journal_entries')
      .where({ account_code: accountCode })
      .orderBy('posting_date', 'desc')
      .limit(limit);
  }

  // Domain-specific: Sum of debits vs credits (must balance)
  async getTrialBalanceTotals() {
    return this.db('journal_entries')
      .where({ status: 'posted' })
      .select(
        this.db.raw('SUM(debit_amount) as total_debits'),
        this.db.raw('SUM(credit_amount) as total_credits')
      )
      .first();
  }

  // Audit trail query
  async getAuditTrail(entityType: string, entityId: string) {
    return this.db('audit_log')
      .where({ entity_type: entityType, entity_id: entityId })
      .orderBy('created_at', 'asc');
  }

  async disconnect(): Promise<void> {
    await this.db.destroy();
  }
}

export class SnapshotManager {
  private snapshots: Map<string, any> = new Map();

  // Capture state before an operation
  async captureSnapshot(label: string, query: () => Promise<any>) {
    this.snapshots.set(label, await query());
  }

  // Compare with current state after an operation
  async compareSnapshot(label: string, query: () => Promise<any>) {
    const before = this.snapshots.get(label);
    const after = await query();
    return {
      before,
      after,
      changes: this.deepDiff(before, after),
      balanceDelta: after?.balance != null
        ? (after.balance - before.balance) : undefined
    };
  }

8.4 Test Data Lifecycle Management

Database test data must be carefully managed to ensure test isolation without corrupting shared environments. The CleanupManager tracks all data created during tests and removes it in reverse order after each scenario.

TypeScript — cleanup-manager.ts
export class CleanupManager {
  private createdRecords: { table: string; id: string; }[] = [];

  // Track records created during test setup
  trackRecord(table: string, id: string): void {
    this.createdRecords.push({ table, id });
  }

  // Teardown: remove all test data in reverse order (respects FK constraints)
  async cleanup(db: DatabaseClient): Promise<void> {
    for (const record of this.createdRecords.reverse()) {
      try {
        await db.query(`DELETE FROM ${record.table} WHERE id = ?`, [record.id]);
      } catch (error) {
        // Log but don't fail — cleanup is best-effort
        logger.warn('Cleanup failed', { table: record.table, id: record.id, error });
      }
    }
    this.createdRecords = [];
  }
}

// In hooks.ts:
Before(async function () {
  this.cleanupManager = new CleanupManager();
  await this.dbClient.connect();
});

After(async function () {
  await this.cleanupManager.cleanup(this.dbClient);
  await this.dbClient.disconnect();
});

Impact: Breaks contract for consumers expecting 2 decimal places.
Suggested Fix: Update the API's decimal serializer to always output 2 decimal places, or update the contract to accept variable precision.

7.4 Self-Healing Capabilities

When API changes cause test failures, the self-healing module compares the previous API specification with the current one and suggests updates to step definitions and feature files.

API Change DetectedAI Suggested Action
Field renamed: accountName → nameUpdate all step definitions referencing accountName. Update JSON schema.
New required field: fiscalYear addedAdd to all test data factories. Update contract definitions. Generate new scenarios.
Endpoint path: /v1/ → /v2/Update base path in config. Run full regression. Generate migration report.
Status changed: 200 → 201 for POSTUpdate expected status codes. Flag for review — may be intentional.

8.3 Reporting Pipeline

TypeScript — Xray Reporter
export class XrayReporter {
  async publishResults(cucumberReport: CucumberJsonReport): Promise<void> {
    const token = await this.authenticate();

    const xrayPayload = {
      testExecutionKey: process.env.XRAY_EXECUTION_KEY,
      info: {
        summary: `GL API Tests - ${env} - ${new Date().toISOString()}`,
        description: `Automated BDD execution | Tags: ${tags}`,
        startDate: this.executionStartTime,
        finishDate: new Date().toISOString(),
        testEnvironments: [process.env.TEST_ENV],
        revision: process.env.GIT_SHA
      },
      tests: cucumberReport.features.flatMap(f =>
        f.elements.map(scenario => ({
          testKey: this.extractXrayKey(scenario.tags),
          status: this.mapStatus(scenario),
          comment: scenario.aiAnalysis || '',
          evidences: this.buildEvidences(scenario),
          steps: scenario.steps.map(step => ({
            status: this.mapStepStatus(step),
            comment: step.result.error_message || '',
            actualResult: step.result.duration + 'ms'
          }))
        }))
      )
    };

    await this.xrayClient.importExecution(xrayPayload);
  }
}

name: GL API Test Suite
on:
  pull_request:
    branches: [main, develop]
  schedule:
    - cron: '0 2 * * *'       # Nightly at 2 AM UTC
  workflow_dispatch:
    inputs:
      environment:
        type: choice
        options: [dev, staging, uat]
      tags:
        description: 'Cucumber tags (e.g. @smoke, @regression)'
        default: '@smoke'

jobs:
  api-tests:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }

      - name: Install dependencies
        run: npm ci

      - name: Run API Tests
        run: |
          npx cucumber-js \
            --tags "${{ inputs.tags || '@smoke' }}" \
            --format json:reports/cucumber-report.json \
            --format html:reports/cucumber-report.html \
            --format @cucumber/pretty-formatter
        env:
          BASE_URL: ${{ secrets[format('API_URL_{0}', inputs.environment)] }}
          AUTH_CLIENT_ID: ${{ secrets.AUTH_CLIENT_ID }}
          AUTH_CLIENT_SECRET: ${{ secrets.AUTH_CLIENT_SECRET }}

      - name: AI Failure Analysis
        if: failure()
        run: npx ts-node src/support/ai-enricher.ts analyze-failures
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}

      - name: Publish to Xray
        if: always()
        run: npx ts-node src/support/xray-reporter.ts publish
        env:
          XRAY_CLIENT_ID: ${{ secrets.XRAY_CLIENT_ID }}
          XRAY_CLIENT_SECRET: ${{ secrets.XRAY_CLIENT_SECRET }}

      - name: Upload Reports
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-reports

SECTION 14

Risk Assessment & Mitigations

RiskProbabilityImpactMitigation
API documentation gaps — incomplete or outdated OpenAPI specs Medium High AI-powered schema discovery from actual responses. Build baselines empirically.
Test environment instability — shared environments with unreliable data High Medium Test data setup/teardown in hooks. Unique identifiers per run. Request dedicated env.
AI cost and latency — LLM API calls adding cost and time Medium Low AI features are opt-in (not critical path). Caching for repeated prompts. Local LLMs (Ollama) option.
Team TypeScript adoption — .NET-focused team learning new stack Medium Medium Onboarding docs. Gherkin is language-agnostic. Steps are simple HTTP calls. Pair programming.
Xray API rate limits — high-volume result publishing Low Low Batch uploads. Retry with exponential backoff. Xray bulk import endpoint.
RabbitMQ test environment access — restricted broker access or shared queues Medium High Use exclusive temporary queues. Request dedicated vhost for test. Implement connection retry logic. Document required AMQP permissions.
Database test data contamination — test data leaking into shared environments Medium High Test prefix convention (TEST-*). Cleanup hooks with reverse FK order. Nightly orphan purge. Read-only assertions where possible.
Message timing / flakiness — async messages arriving late causing false failures High Medium Configurable timeouts per scenario. Polling with backoff. Separate async tests from sync tests in tag strategy. Retry on timeout.
Database schema changes — column renames or table restructuring breaking queries Medium Medium Migration checker validates expected schema version before tests run. AI suggests query updates when schema changes detected.
Schema/Contract maintenance — keeping definitions in sync High Medium Automated drift detection. AI suggests schema updates. Contract review in PR process.
